# Conventions

## Shell scripts

- Use bash strict mode in every shell script: `#!/usr/bin/env bash` and `set -euo pipefail`.
- Write feature installers as entrypoint scripts that run as root.
- Follow the repository’s standard structure for each script type and keep related scripts consistent in filenaming, layout, style, sections, conventions and patterns used.
- All log messages, including output and error messages, should be handled consistently for all script instances.
- Separate section headers from ordinary comments with a blank line.
- Place comments that describe a code block immediately above that block.
- Prefer small, focused helpers and keep logic readable rather than over-abstracting.
- Do not use TRAP or defensively program for unlikely/rare conditions.
- Perform explicit checks for option inputs, compatibility, prerequisites, outputs etc as required or appropriate for the specific type/class of script.
- To mitigate the risk of poisoned packages, npm, pip, pipx installations use their equivalent "minimum release age" options with a default value of seven days where this option is to be configurable.
- Use lowercase `snake_case` for shell variables.

## Installer structure

- Keep feature installer scripts in the same section order and headings:
  1. Inputs
  2. Collect options
  3. Logging functions
  4. Check option compatibility
  5. Prerequisites
  6. Check required commands
  7. Installation
  8. Install functions
  9. Install packages
- Read feature options once into normalized local variables before use.
- Consume options from uppercase environment variables that are populated from `devcontainer-feature.json`.
- Make each declared option change installer behavior. If a capability cannot be implemented by the feature specification, keep it in the consuming `devcontainer.json` instead.

## Shared helpers and package installation

- Use the shared `log`, `error`, and `require_command` helpers unless a script has a clear reason to do otherwise.
- Installers that invoke `apt` directly should use the shared `install_apt_packages` helper.
- `install_apt_packages` should perform one complete apt cycle: `apt-get update`, `apt-get install -y --no-install-recommends`, `apt-get clean`, and removal of `/var/lib/apt/lists/*`.
- Fail early with a clear message when an optional capability requires a missing prerequisite.
- Use npm global installs with `--global --ignore-scripts --min-release-age`.
- Use constrained pipx installs with pip’s `--uploaded-prior-to` argument.

## Feature metadata and dependencies

- Expose feature options through `devcontainer-feature.json` and consume them in `install.sh`.
- Keep feature options limited to behavior the Feature specification can implement.
- Treat host mounts, user configuration, and similar host-side concerns as the responsibility of the consuming `devcontainer.json`.
- Reserve `installsAfter` for portable, published-feature dependencies.
- For local test composition in this repository, use `overrideFeatureInstallOrder` when a local Node feature must be installed before npm-dependent features.

## Tests and scenarios

- Keep test scripts in the same section order and headings:
  1. Inputs
  2. Prerequisites
  3. Tests
- Use `source dev-container-features-test-lib` in test scripts before running checks.
- Prefer concrete assertions that verify real behavior and call `reportResults` at the end.
- Use `ubuntu:latest` for scenario and integration coverage.
- When a scenario depends on Node.js, compose the local Node feature explicitly in the test setup.
- For autogenerated isolated tests, retain a Node-equipped base image where the Dev Container CLI cannot express that composition.

## Documentation and maintenance

- Keep scripts, tests, and documentation aligned with the current repository conventions and the actual behavior of the feature.
- Update README and related notes when behavior, options, or prerequisites change.
- Prefer comments that explain intent or non-obvious decisions rather than restating obvious code.
